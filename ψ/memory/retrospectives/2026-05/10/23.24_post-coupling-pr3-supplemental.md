## Lessons Learned

1. **Don't /rrr until inbox is genuinely silent** — not when *you* feel done. /rrr is a snapshot, not a curtain call. Better trigger: "no pending inbox + no blocked-on-me threads + explicit close signal from human/coordinator". If a routed task lands after /rrr, write a supplemental retro — don't pretend it didn't happen. Today: 22:44 /rrr, 22:57 John's coupling check arrived, +3 maw rounds + 1 PR happened before next /rrr. The first /rrr was a *checkpoint*, not a *wrap*.

2. **Engineering replies use numbers + standards citations, not opinions** — John asked 7 OQs across HVAC + BAS + acoustics. Forge answered each with: a numeric value (475 GPM @ 150 kPa), a standards reference (ASHRAE 90.1-2022 §6.5.4.2), or a specific product (Contemporary Controls BASrouter ~$300). No "I think" / "probably" / "depending on…". Numbers close questions; opinions invite ping-pong. Pattern reusable: if you're answering a peer engineer, the answer should be either (a) a number with units, (b) a citation, (c) "we don't know yet" — never (d) a vibe.

3. **Commit strategy = logical groups, not one big bag** — same 11 files could have been one commit ("day 2 work"). Three commits read better because each tells a coherent story: housekeeping (fleet comms, untracked legacy), substance (spike prep, the actual deliverable), meta (retro/lesson, the process). Conventional commit prefixes (`chore` / `feat` / `docs`) signal the category before the reader opens the diff. Reviewer attention is the scarcest resource — structure earns it.
